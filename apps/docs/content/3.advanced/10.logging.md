## 两个通道，一个 Composable

`useMcpLogger()` 提供了一个 **分离通道** API，因为这两个目标受众非常不同：

| Channel | Audience | API |
|---------|----------|-----|
| **Client notifications** (`notifications/message`) | 终端用户 / 代理 UI（Cursor、Claude、MCP Inspector，…） | `log.notify(...)`, `log.notify.debug/info/warning/error(...)` |
| **Wide events** (server-side, powered by [evlog](https://evlog.dev)) | 运维人员、开发终端、drains（Axiom、OTLP、Datadog，…） | `log.set(...)`, `log.event(...)`, `log.evlog` |

Notifications 是 **面向用户** 的，可能最终出现在聊天记录中。Wide events 是 **面向运维人员** 的，会在每次请求结束时以美观的格式打印到开发终端，并在生产环境中发送到 drains。

::code-collapse

```txt [Prompt]
向我的 Nuxt MCP 服务器（@nuxtjs/mcp-toolkit）添加结构化日志。

- 在工具处理器内部使用 useMcpLogger()（自动导入）
- log.notify.info({ … }) 会向已连接的 MCP 客户端发送 notifications/message
- 快捷方式：log.notify.debug/info/warning/error — 遵循 logging/setLevel
- log.set({ user: { id } }) 会将上下文累积到请求的 evlog wide event 中
- log.event('charge_started', { amount }) 会在同一个 wide event 中捕获一个离散事件
- log.evlog 提供底层的 RequestLogger（fork、error、getContext，…）
- 安装可选的 evlog peer 依赖以启用 wide events：`pnpm add evlog`
- 通过一个 Nitro 插件将内容发送到 Axiom / Sentry / Datadog / OTLP / HyperDX / Better Stack / PostHog（`server/plugins/evlog-axiom.ts`）
- 自定义 drains：在 `evlog:drain` hook 上注册任意 `(ctx) => Promise<void>`
- 通过 mcp.logging: false 完全禁用可观测性（notify 仍然可用）

文档：https://mcp-toolkit.nuxt.dev/advanced/logging
```

::

## 设置

`log.notify(...)` 开箱即用——无需额外设置。服务器端的 wide events 由可选的 [evlog](https://evlog.dev) peer 依赖提供支持。

### 启用 wide events

将 evlog 与 toolkit 一起安装：

```bash
pnpm add evlog
```

就是这样。toolkit 会自动检测 evlog 并将其自动接入 Nitro——每次 MCP 请求都会在开发终端中显示 wide events，且已经标记好 `mcp.tool`、`mcp.session_id` 等字段。

### 配置（可选）

在 `mcp.logging` 下转发 [evlog Nitro options](https://evlog.dev)：

```typescript [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@nuxtjs/mcp-toolkit'],
  mcp: {
    logging: {
      // 在这里转发任意 evlog/nitro 选项（drains、sampling、redaction，…）
    },
  },
})
```

### 强制开启 / 选择退出

| `mcp.logging` value | Behavior |
|---|---|
| `undefined`（默认） | 自动检测：如果安装了 `evlog` 则开启，否则关闭。 |
| `true` 或对象 | 强制开启。如果缺少 `evlog`，构建会抛出并附带安装说明。 |
| `false` | 强制关闭。`log.notify(...)` 仍可工作；`log.set(...)` / `log.event(...)` / `log.evlog` 会抛出 `McpObservabilityNotEnabledError`。 |

## `useMcpLogger()`

自动导入。必须在 tool、resource 或 prompt 处理器内部调用。

```typescript [server/mcp/tools/charge.ts]
import { z } from 'zod'
import { defineMcpTool } from '@nuxtjs/mcp-toolkit/server'

export default defineMcpTool({
  name: 'charge_card',
  description: '扣款支付方式',
  inputSchema: {
    userId: z.string(),
    amount: z.number().int(),
  },
  handler: async ({ userId, amount }) => {
    const log = useMcpLogger('billing')

    // → server: 合并到请求的 wide event 中，在请求结束时
    //   打印到开发终端，并发送到 drains。
    log.set({ user: { id: userId }, billing: { amount } })

    // → client: 显示在 MCP Inspector 的“Server Notifications”面板中
    //   （以及 Cursor / Claude 的日志查看器中）。遵循 `logging/setLevel`。
    await log.notify.info({ msg: '开始扣款', amount })

    try {
      const receipt = await charge(userId, amount)
      log.event('charge_completed', { receiptId: receipt.id })
      await log.notify.info({ msg: '扣款成功', receiptId: receipt.id })
      return `已扣款 ${amount}。`
    }
    catch (err) {
      log.evlog.error('扣款失败', err)
      await log.notify.error({ msg: '扣款失败', error: String(err) })
      throw err
    }
  },
})
```

发生的事情：

1. MCP 客户端会收到 **两条 `notifications/message`** 记录（`info`、`info`），如果扣款失败则还会收到一条 `error`。
2. 该请求的 wide event 会被丰富为 `user.id`、`billing.amount`，以及一个 `charge_completed`（或 `charge_failed`）离散事件——可在开发终端中看到，并转发到任何已配置的 evlog drain。

### API

| Method | Channel | Description |
|--------|---------|-------------|
| `notify(level, data, logger?)` | client | 发送一个 `notifications/message`。如果级别被客户端过滤，则静默丢弃。 |
| `notify.debug(data, logger?)` | client | `notify('debug', …)` 的快捷方式。 |
| `notify.info(data, logger?)` | client | `notify('info', …)` 的快捷方式。 |
| `notify.warning(data, logger?)` | client | `notify('warning', …)` 的快捷方式。 |
| `notify.error(data, logger?)` | client | `notify('error', …)` 的快捷方式。 |
| `set(fields)` | server (evlog) | 将字段合并到当前请求的 wide event 中。 |
| `event(name, fields?)` | server (evlog) | 在 wide event 的请求日志中捕获一个离散事件。 |
| `evlog` | server (evlog) | 底层的 [`RequestLogger`](https://evlog.dev/docs/api/request-logger) — `fork`、`error`、`getContext`、`emit`，… |

`logger` 是附加到通知上的前缀（客户端可见）。将其作为 `useMcpLogger('prefix')` 的参数传入，以便为请求设置默认值，或在单次调用中传入以覆盖默认值。

### 级别过滤

根据 MCP 规范，客户端可以调用 `logging/setLevel` 来筛选他们想接收哪些消息。toolkit 会把当前 MCP session id 转发给 SDK，因此每个 session 的过滤规则都会独立应用——对于请求 `warning` 或更高级别的客户端，`log.notify.debug(...)` 会变成无操作。

即使在传输断开或级别被过滤的情况下，`notify` 方法也始终会 resolve 且不会抛出，因此你可以放心在高频路径中自由使用。

## 使用 evlog 的 Wide Events

每个 MCP 请求都会被一个 evlog **wide event** 包裹——每个请求一条结构化日志，随着处理器执行而累积。toolkit 会自动为每个 wide event 添加以下标签：

| Field | Description |
|-------|-------------|
| `mcp.transport` | `streamable-http`（默认）或 Workers 上的 `cloudflare-do` |
| `mcp.route` | 配置的 MCP 端点路径 |
| `mcp.session_id` | 从 `mcp-session-id` header 复制而来（如果存在） |
| `mcp.method` | JSON-RPC 方法（`tools/call`、`tools/list`、`initialize`、`resources/read`、`prompts/get`、`notifications/initialized`，…） |
| `mcp.request_id` | 当消息包含时对应的 JSON-RPC request id |
| `mcp.tool` | `tools/call` 请求中的工具名称 |
| `mcp.resource` | `resources/read` 请求中的 URI |
| `mcp.prompt` | `prompts/get` 请求中的 prompt 名称 |

对于批量 JSON-RPC 载荷，单数键（`method`、`tool`、`resource`、`prompt`、`request_id`）会切换为复数数组（`methods`、`tools`、`resources`、`prompts`、`request_ids`）。不需要用户代码或中间件——这些字段会自动落到每个请求上。

你可以使用 `set()` 叠加额外上下文，并用 `event()` 捕获里程碑：

```typescript [server/mcp/tools/import-csv.ts]
import { defineMcpTool } from '@nuxtjs/mcp-toolkit/server'

export default defineMcpTool({
  name: 'import_csv',
  description: '导入 CSV 文件',
  inputSchema: { url: z.string().url() },
  handler: async ({ url }) => {
    const log = useMcpLogger('import')
    log.set({ source: { kind: 'csv', url } })

    const rows = await fetchCsv(url)
    log.event('rows_fetched', { count: rows.length })

    const inserted = await insertRows(rows)
    log.set({ result: { inserted } })

    return `已导入 ${inserted} 行。`
  },
})
```

在开发终端中，你会看到类似这样的内容：

```bash
INFO [Playground MCP] POST /mcp 200 in 18ms
├─ requestId: c6a9094f-bd24-4971-88ee-a53a28fab13a
├─ mcp: transport=streamable-http route=/mcp session_id=ef39698f-… method=tools/call request_id=1 tool=import_csv
├─ source: kind=csv url=https://example.com/data.csv
├─ result: inserted=128
└─ requestLogs: 0={"level":"info","message":"rows_fetched",...}
```

## 将内容发送到你的可观测性栈

evlog 为 Axiom、Sentry、OpenTelemetry / OTLP、HyperDX、Datadog、Better Stack 和 PostHog 提供了 **drain 适配器**。drain 本质上就是在 `evlog:drain` Nitro hook 上注册的一个函数——每个 MCP wide event（已经带有 `mcp.tool`、`mcp.session_id`，…）都会被转发到你接入的目标。

模式始终相同：在 `server/plugins/` 下放置一个 Nitro 插件，导入 `evlog/adapters/*` 中的 `createXxxDrain()`，并把它注册到该 hook 上。大多数适配器如果你设置了标准环境变量，都是 **零配置** 的。

```typescript [server/plugins/evlog-axiom.ts]
import { createAxiomDrain } from 'evlog/adapters/axiom'

export default defineNitroPlugin((nitroApp) => {
  nitroApp.hooks.hook('evlog:drain', createAxiomDrain())
})
```

```bash [.env]
NUXT_AXIOM_TOKEN=xaat-...
NUXT_AXIOM_DATASET=mcp-server
```

现在你可以在 Axiom 中按 `mcp.tool`、`mcp.session_id`、`user.id`、latency、error rate 来切分 MCP 流量——你在 wide event 上通过 `set()` 设置的每个字段都可以被查询。

这个 hook 是可叠加的：可以注册多个 drains 并行转发（例如 Axiom 用于存储 + Sentry 用于告警），而一个 drain 的失败不会影响其他 drain。

```typescript [server/plugins/evlog-drains.ts]
import { createAxiomDrain } from 'evlog/adapters/axiom'
import { createSentryDrain } from 'evlog/adapters/sentry'

export default defineNitroPlugin((nitroApp) => {
  nitroApp.hooks.hook('evlog:drain', createAxiomDrain())
  nitroApp.hooks.hook('evlog:drain', createSentryDrain())
})
```

drain 只是 `(ctx: DrainContext | DrainContext[]) => Promise<void>`——你可以自己编写它，用于在 `tools/call` 失败时发送 Slack 告警、持久化到自己的数据库以便审计、转发到内部 SIEM，等等。

::callout{icon="i-lucide-arrow-up-right" to="https://evlog.dev" target="_blank"}
查看 [evlog 文档](https://evlog.dev) 以了解完整的适配器列表、配置选项（sampling、redaction、enrichers），以及如何构建自定义 drains。
::

## 生产建议

- **谨慎使用 `notify`。** 每条通知都会产生一次网络往返，并最终出现在用户的聊天历史中。请仅在可操作的进度更新（“已打开 PR #42”）或真正的错误时使用它。
- **大量使用 `set` 和 `event`。** 它们成本低，会批量合并到宽事件中，而且不会泄露给客户端。你可以在这里放置结构化业务数据、耗时、重试次数等。
- **与 `extractToolNames()` 配合使用。** 在中间件中将 `useMcpLogger()` 与 [`extractToolNames(event)`](/advanced/middleware) 结合起来，记录每一次工具调用，包括失败的调用，用于审计追踪。

## 要求

::callout{icon="i-lucide-triangle-alert" color="warning"}
`useMcpLogger()` 需要 `nitro.experimental.asyncContext` 为 `true`（自 Nuxt 3.8+ 起默认如此）。当可选的 `evlog` 依赖缺失或 `mcp.logging` 被设置为 `false` 时，`set`、`event` 和 `evlog` 通道会抛出 `McpObservabilityNotEnabledError`。`notify` 通道在这两种情况下都能继续工作。
::
